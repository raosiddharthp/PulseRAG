PulseRAG: Grounding LLMs
in live enterprise data.

Traditional RAG indexes documents. PulseRAG queries the operational nervous system of an enterprise — ERP records, CRM pipeline, ticketing systems, live inventory — at the moment a question is asked, then synthesises the answer across systems that don't talk to each other.

Abstract
Enterprise teams routinely ask questions whose answers are distributed across multiple systems of record — CRM pipelines, ERP fulfilment data, IT ticketing queues — with no single system capable of answering them. Traditional RAG addresses document retrieval latency; it does not address cross-system synthesis. PulseRAG addresses the gap directly: a natural language query layer that classifies intent, decomposes queries into parallel sub-queries per system, federates across heterogeneous enterprise connectors, and renders responses in the format the answer demands — table, chart, card, or prose. The system is designed for a free-tier infrastructure stack, with Groq (Llama 3.1 8B) as the inference layer and seeded mock connectors for demonstration. All architectural tradeoffs are documented as formal ADRs.

§ 01 Motivation

The joins that
don't exist anywhere.

Between 2022 and 2025, enterprise teams broadly adopted RAG as a solution to LLM hallucination. The implementation pattern is largely uniform: documents are chunked, embedded into a vector store, and retrieved via cosine similarity against an embedded query. This handles document-level knowledge well.

It does not handle operational questions. A Sales Director asking "which at-risk deals have fulfilment issues that could affect close?" needs an answer that lives in three systems simultaneously — the CRM deal stage, the ERP fulfilment status, and the ticketing queue for account friction. No vector store contains that join. No single system of record can answer it.

The gap is not a retrieval problem. It is a federation problem. The answer exists — distributed across systems that were never designed to talk to each other, requiring a human to spend 30–45 minutes context-switching across three logins to assemble it manually.

§ 02 Problem Statement

What traditional RAG
cannot answer.

The failure mode of document RAG on operational enterprise questions is not noisy — it is silent. The system returns an answer with high confidence. The answer is technically grounded in a document. The document is simply the wrong unit of truth for the question being asked.

Operational enterprise questions require current state from systems of record, not historical text from indexed documents. The table below maps the failure pattern across the query types PulseRAG is designed to handle.

The worked failure example

A Sales Director queries: "Which of our at-risk deals this quarter have fulfilment issues that could affect close?"

A document RAG system has no document that answers this. The nearest indexed content might be a sales playbook or a process guide — both returned with high cosine similarity, both factually correct, both completely wrong for the question. The generation layer produces a fluent, confident, and useless answer.

PulseRAG routes this to three connectors simultaneously: CRM for deal stage and risk flags, ERP for fulfilment and inventory status, Ticketing for open account friction. The synthesis layer joins the results and renders a ranked deal table — the format the answer actually demands.

§ 04 ADRs

Architecture Decision
Records.

Every significant architectural choice is documented below with status, context, decision rationale, consequences, and alternatives considered. Following TOGAF ADM principles — design precedes build, decisions are traceable.

ADR-001
Groq (Llama 3.1 8B) as LLM inference layer
Accepted
Context
LLM inference is required for intent classification and response synthesis. Claude and GPT-4o APIs require paid accounts. Budget is zero.
Decision
Use Groq free tier with Llama 3.1 8B. Near-zero latency, no credit card, sufficient instruction-following for structured output.
Consequences
Rate-limited at high volume. Weaker instruction-following than GPT-4o for edge cases. Acceptable for a controlled demo scenario set.
Alternatives
Claude API (paid), GPT-4o-mini (paid), HuggingFace Inference API (free but slower, cold starts).
ADR-002
Direct browser-side API calls vs. backend proxy
Accepted
Context
PulseRAG must call the Groq API from a static HTML page with no backend infrastructure.
Decision
Call Groq directly from the browser via fetch(). Groq's free tier permits CORS requests from browser origins.
Consequences
API key is visible in client code — acceptable for a portfolio demo, not for production. In production, a backend proxy would be mandatory.
Alternatives
FastAPI proxy on Railway (free tier) — adds cold start latency and operational complexity.
ADR-003
Mock connectors over real enterprise integrations
Accepted
Context
Demonstrating real Salesforce/SAP/Jira connectors requires enterprise credentials, sandbox environments, and OAuth flows.
Decision
Implement mock connectors as seeded JSON objects with realistic schemas — same field names, same data relationships, same response structure as real APIs.
Consequences
Data is not live. Demonstrates architectural intelligence, not integration depth. Connector interface is designed to be real-API-swappable.
Alternatives
Salesforce Developer Edition (free) — viable for CRM only; ERP and Ticketing have no equivalent free sandbox.
ADR-004
Scenario-scoped query matching for design phase
Accepted · Design phase only
Context
The design-phase simulator must run without Groq API calls, demonstrating the full pipeline visually.
Decision
Pre-script four scenario responses with keyword-triggered routing. Groq is wired in the build phase for arbitrary query handling.
Consequences
Design simulator does not handle arbitrary queries. Out-of-scope queries return a graceful fallback. Fully replaced by Groq in build.
Alternatives
Groq in design phase — requires API key in static HTML at design time; deferred to build phase by deliberate design.
ADR-005
Adaptive output format selection by intent
Accepted
Context
Natural language responses to cross-system queries are insufficient at scale — a ranked list of 40 deals as prose is unusable.
Decision
LLM response includes a format signal (TABLE | CHART | CARD | PROSE). Render layer reads the signal and delegates to the appropriate component.
Consequences
Format inference can fail on ambiguous queries — fallback to PROSE. LLM prompt must explicitly instruct format signal output.
Alternatives
Rule-based format selection (regex on query) — fragile; LLM signal is more robust and extensible.
ADR-006
In-memory TTL cache over Redis
Accepted · Demo scope
Context
Cache layer is needed to demonstrate the freshness policy concept without hitting connectors on every query.
Decision
Implement cache as a JavaScript Map with per-entry TTL timestamps. Checked on every connector call before dispatching.
Consequences
Cache does not persist across sessions or browser tabs. No distributed invalidation. Production would require Redis or equivalent.
Alternatives
Redis on Railway free tier — adds infrastructure complexity without material demo benefit.
ADR-007
Static single-file HTML delivery
Accepted
Context
Portfolio page must be hostable on GitHub Pages with zero build infrastructure and zero cost.
Decision
Single index.html — all CSS, JS, SVG diagrams, simulator logic, and ADRs self-contained. No bundler, no framework, no build step.
Consequences
File grows large. No component separation. Acceptable for a portfolio artifact — prioritises deliverability over maintainability.
Alternatives
React + Vite on Vercel — unnecessary complexity for a static portfolio document.
ADR-008
Freshness policy as a query-type routing layer
Accepted
Context
Not all enterprise queries require live data. Hitting the source on every request burns API quota and increases latency unnecessarily.
Decision
Maintain a TTL policy map keyed by query type. High-sensitivity types (inventory, live tickets) bypass cache. Low-sensitivity types (account info) use TTL.
Consequences
Staleness risk on low-sensitivity queries remains. Policy map must be maintained as query types evolve. Surfacing data age to UI mitigates user trust risk.
Alternatives
Always-live (no cache) — correct but expensive and high-latency at scale. Always-cached — cheap but risks operational decisions on stale data.
